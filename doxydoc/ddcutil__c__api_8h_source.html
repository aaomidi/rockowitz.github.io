53 /* Note on "report" functions.
54  *
55  * Various functions with "report" in the name, e.g. ddca_report_display_ref(),
56  * ddca_report_display_info_list(), write formatted reports to (normally) the
57  * terminal. Sometimes, these are intended to display data structures for
58  * debugging. Other times, they are used to format output for the ddcutil
59  * command line program.
60  *
61  * The operation of these functions can be tweaked in two ways.
62  * - The "depth" parameter is a logical indentation depth. This allows
63  * reports that invoke other reports to perform indent the subreports
64  * sensibly. At the level of the ddcutil_c_api(), one unit of
65  * logical indentation depth translates to 3 spaces.
66  * - The destination of reports is normally the STDOUT device. This can
67  * be changed by calling set_fout().
68  */

169 // I2C is an inherently unreliable protocol. The application is responsible for
170 // retry management.
171 // The maximum number of retries can be tuned.
172 // There are 3 retry contexts:
173 // - An i2C write followed by a read. Most DDC operations are of this form.
174 // - An I2C write without a subsequent read. DDC operations to set a VCP feature value
175 // are in this category.
176 // - Some DDC operations, such as reading the capabilities string, require multiple
177 // write/read exchanges. These multi -part exchanges have a separate retry count
178 // for the entire operation.
